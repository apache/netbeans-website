<div class="sect2">
<h3 id="_what_is_a_background_thread_and_why_do_i_need_one">What is a background thread and why do I need one?</h3>
<div class="paragraph">
<p>As with most user interface (UI) toolkits, Swing is <em>single threaded</em>.  That means there is one and only one thread that should create or alter the state of UI components, and that is the AWT Event Dispatch Thread (also known as the EDT or the "event thread").  It processes things like key and mouse events and calls components to respond to them.</p>
</div>
<div class="paragraph">
<p>This also means that code that responds to a key or mouse event, or some call triggered by one, should run very quickly, because the user can be typing or clicking, but the entire application is <em>blocked</em> from responding to more events until your code exits.  So sometimes you will want to move expensive or slow work onto a <a href="DevFaqThreading.asciidoc">background thread</a>.</p>
</div>
<div class="paragraph">
<p>A background thread is any thread that is not the event thread.</p>
</div>
<div class="paragraph">
<p>If you are running on some background thread, but need to modify some Swing component, a simple way to do this is &lt;pre&gt;</p>
</div>
<div class="listingblock">
<div class="content">
<pre class="prettyprint highlight"><code class="language-xml" data-lang="xml">   EventQueue.invokeLater(new Runnable() {
     public void run() {
       //this code can work with Swing
     }
   });&lt;/pre&gt;</code></pre>
</div>
</div>
<div class="paragraph">
<p>Note that the caveat about Swing includes creating components - it is probably not safe to even construct Swing components on a background thread, because of synchronization on <code>Component.getTreeLock()</code>.</p>
</div>
</div>
