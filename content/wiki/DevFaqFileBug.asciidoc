<h3 id="_filing_a_bug_report">Filing a bug report</h3>
<div class="paragraph">
<p>If you think you have found a bug in the NetBeans Platform or IDE which affects your module development, please file it so it can be fixed. Generally exceptions coming from platform code are bugs in NetBeans (unless it is e.g. an <code>IllegalArgumentException</code> thrown after your code called a method with invalid arguments). Other things can of course be bugs if NetBeans is not behaving according to its documentation, or if something just looks wrong.</p>
</div>
<div class="olist arabic">
<ol class="arabic">
<li>
<p>Reread all relevant documentation to see if you have missed anything important.</p>
</li>
<li>
<p>If you are unsure whether the behavior is really incorrect, you can ask on <code>dev@platform.netbeans.org</code>. Do not be too shy to file a bug, though; it is just as easy to close an invalid bug report as it is to reply to the mailing list.</p>
</li>
<li>
<p>If at all possible, figure out how to reproduce your bug. From scratch: the assignee of the bug report cannot see your computer and has no idea what you are working on or why. Try to make a <em>minimal</em>, self-contained test case that anyone could run to see the bug in action. Often a suite project is a good test case - attach a ZIP of sources, including nested module projects. If you know how to write a unit test for the buggy module, that is ideal, but this can require some deeper knowledge of NetBeans internals. Sometimes a bug occurs that just cannot be easily reproduced - it is still fine to file a bug, but include as much diagnostic information as you can and do not be surprised if it does not get fixed.</p>
</li>
<li>
<p>Read: <a href="IssueReportingGuidelines.asciidoc">Issue Reporting Guidelines</a></p>
</li>
<li>
<p>For general background you may also want to read: <a href="http://www.catb.org/esr/faqs/smart-questions.html">How To Ask Questions The Smart Way</a></p>
</li>
<li>
<p>File a bug report and include at least</p>
</li>
<li>
<p>Some background on what you are trying to accomplish and why.</p>
</li>
<li>
<p>Some kind of test case to demonstrate the bug.</p>
</li>
<li>
<p>Instructions for running the test case.</p>
</li>
<li>
<p>What you would expect to see happen.</p>
</li>
<li>
<p>What you actually see happen.</p>
</li>
<li>
<p>Be patient as the bug is assigned and evaluated, and provide additional information if requested. If all goes well it should be fixed for a future NetBeans release. The evaluator may also be able to offer some workarounds for use in current releases.</p>
</li>
</ol>
</div>
