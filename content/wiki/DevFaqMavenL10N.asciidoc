<div class="sect1">
<h2 id="_how_can_i_create_localization_modules_using_maven">How can I create localization modules using Maven?</h2>
<div class="sectionbody">
<div class="paragraph">
<p>If you have localized versions of resources <em>in your own modules</em>, you can simply place them alongside the base version; for example, <code>src/main/resources/my/module/Bundle_de.properties</code> would provide a German translation of <code>src/main/resources/my/module/Bundle.properties</code>.</p>
</div>
<div class="paragraph">
<p>But to localize <em>other people&#8217;s modules</em>, then as explained in <a href="http://bits.netbeans.org/dev/javadoc/org-openide-modules/org/openide/modules/doc-files/i18n-branding.html#placement">Physical Placement of Localized Resources</a> you will need to create <code>modules/locale/<strong>_</strong>.jar</code> JARs with the localized resources. (Simply bundling localized resources in your own module will not work, because these are usually searched for starting with the original module&#8217;s <code>ClassLoader</code>, which cannot "see" your modules.)</p>
</div>
<div class="paragraph">
<p>The simplest way to do this is to use the existing [<a href="http://mojo.codehaus.org/nbm-maven-plugin/branding-mojo.html">http://mojo.codehaus.org/nbm-maven-plugin/branding-mojo.html</a> <code>nbm:branding</code> goal], which despite its name can handle localization as well as branding. (These are nearly interchangeable in the NB Platform.) <span class="image"><img src="Localedemo.zip" alt="Localedemo"></span> demonstrates how to localize parts of the Options dialog to Czech. Take note of:</p>
</div>
<div class="ulist">
<ul>
<li>
<p><code>lokalizace/pom.xml</code> asks to use this goal when building the module. It overrides the "branding token" (otherwise inherited from the parent POM) with your locale identifier. (Specifying this custom execution is required in the 3.3 plugin; future versions may run the goal automatically during the package phase when there is anything to process, but you would still need to declare the locale in your plugin configuration.)</p>
</li>
<li>
<p><code>lokalizace/src/main/nbm-branding/</code> contains the actual localizations.</p>
</li>
<li>
<p>Localizations must be grouped by module to be localized, under a folder path matching the path of the original module in its cluster.</p>
</li>
<li>
<p>The locale suffix (<code>_cs</code> in this case) is inserted automatically by the build, so the source file should be named simply <code>Bundle.properties</code>.</p>
</li>
<li>
<p><code>application/pom.xml</code> specifies <code>--locale cs</code> so that you can quickly test the localization just by running the app.</p>
</li>
<li>
<p>It also turns on <code>NbBundle</code> debugging mode when requested by a special profile. This can be helpful to pinpoint where a particular English string is coming from: every visible string will be annotated with a bundle number (cross reference with console output) and line number. You can also use the Search field in the branding editor (available from the context menu of <code>lokalizace</code>) to find all potentially localizable strings in the Platform matching some keyword; this editor also creates the right folder structure for you automatically when localizing new bundles.</p>
</li>
</ul>
</div>
<div class="paragraph">
<p>The module bundling the localizations can now be included in your app, or distributed by any other means. When the specified locale is active, your localizations will be applied.</p>
</div>
</div>
