<div class="sect2">
<h3 id="_why_use_this_strange_convention_wouldn_t_a_map_be_good_enough">Why use this strange convention - wouldn&#8217;t a Map be good enough?</h3>
<div class="paragraph">
<p>Other platforms do use string-keyed maps for this sort of thing, but there are some weaknesses with that approach:</p>
</div>
<div class="olist arabic">
<ol class="arabic">
<li>
<p>it is impossible to enforce <a href="DevFaqModuleDependencies.asciidoc">dependencies</a>.  With <code><a href="DevFaqLookup.asciidoc">Lookup</a></code>, a module&#8217;s code cannot request an object of type X unless it can load/access the <code>X.class</code>. The module trying to look up an X will not be able to see <code>X.class</code> unless it <a href="DevFaqModuleDependencies.asciidoc">declares a dependency</a> on the module that defines X <em>and</em> the module which defines X says that it allows access to the Java package X lives in. (A <code>Map&lt;Class&lt;T&gt;,T&gt;</code> would do the same job as <code>Lookup</code>.)</p>
</li>
<li>
<p>The class of values in a map can change without notice - so if you have <code>(SomeIface) foo = (SomeIface) globalMap.get("foo")</code>, some new version of the module that provides "foo" can change the return type, causing <code>ClassCastException</code> s;  with <code>Lookup</code>, you cannot ever get an object that is not of the type you passed in the request - so Lookup&#8217;s approach is more robust.</p>
</li>
<li>
<p><code>Lookup</code> supports listening to changes in the result.</p>
</li>
<li>
<p><code>Lookup</code> supports multiple instances for one key - if you call <code>lookup(X.class)</code> you get one instance.  If you call <code>lookupAll(X.class)</code> you get a <code>Collection&lt;? extends X&gt;</code> (so with <code>lookupAll()</code> it is more like a <code>Map&lt;Class&lt;T&gt;,List&lt;T&gt;&gt;</code>)</p>
</li>
</ol>
</div>
<div class="paragraph">
<p>There are some other capabilities of <code>Lookup</code> (such as getting the specific type or number of results without actually creating objects, and providing named result items) but these are rarely used in practice.</p>
</div>
<div class="paragraph">
<p><code>Lookup</code> is very powerful, yet simple and generic;  people quickly learn to love it, once they realize what it can do.</p>
</div>
<div class="paragraph">
<p>See also the javadoc: [<a href="http://bits.netbeans.org/dev/javadoc/org-openide-util-lookup/org/openide/util/Lookup.html">http://bits.netbeans.org/dev/javadoc/org-openide-util-lookup/org/openide/util/Lookup.html</a> <code>org.openide.util.Lookup</code>]</p>
</div>
</div>
